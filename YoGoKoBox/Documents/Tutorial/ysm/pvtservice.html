    Position, Velocity and Time (Pvt)

    The Pvt service manages positioning, movement and time information of the station.

    It can manage sources of different types (GNSS source, NTP, static positioning, etc) and merge data.

    Processing and I/O

    The positioning system is split into several services:

    • Positioning provider services (StaticProvider, GpsProvider, NtpProvider, ...) provide data from a specific source to the Pvt service through publish-subscribe messages. The AutotalksService also embeds positioning capabilities and thus acts as a positioning provider service to the Pvt service.
    • The Pvt service is in charge of managing sources and merging data in order to generate the fusioned positioning data. It provides the fusioned data in the VDP cache for access by other services.
    • The PathHistoryManager service generates the vehicle Path History from data available in the VDP cache.
    PvtService

    Groups of data and source profiles

    Sources does not all provide the same informations. For instance, some sources only provide timing informations (ntp), or do not provide details about leap seconds (gps).

    The Pvt service distinguishes the following groups of data:

    • time: Time information (utctime or itstime) and error
    • ls2004: Leap seconds since 2004
    • latlon: Latitude, longitude and errors
    • alt: Altitude and error
    • speed: Speed and error
    • heading: Heading and error
    • climb: Vertical speed and error

    Each source is attributed a profile, stating what is the format of a GNSS fix from this source and which groups of data it supports:

    • autotalks: Feed from Autotalks (JSON or MessagePack), all fields are provided
    • gpsd: Feed from gpsd (JSON), all fields except ls2004 are provided
    • native: Feed using native data (JSON), all fields are provided
    • time: Feed using UTC time data (JSON), only time is provided
    • user: Feed using user data (JSON), all fields are provided

    The list of fields that are actually retrieved from the source may then be restricted by configuration.

    Merging strategy

    A basic merging strategy is used: Sources are sorted by priority, according to their order in the configuration (source1 is given the highest priority). For each field, Pvt takes the field from the source which has the highest priority. A source is also discarded when it is disabled or when it reaches a timeout condition (i.e. when this source did not send a fix during a given amount of time).

    As a summary, for each group of fields, the Pvt service considers data from the source of the greatest priority that is neither disabled or timed out.

    The following table describes an example of merging:

    n. Source Time Ls2004 LatLon Alt. Speed Verdict
    1 User input(timedout) - - - - - -> Ignored because timed out
    2 Autotalks (disabled) 1421231234 6 48.1;-1.6 200 25 -> Ignored because disabled
    3 Static LS2004 - 5 - - - -> Take Ls2004
    4 Ntp 1559876543 - - - - -> Take Time
    5 Gpsd 1555675678 - 48.1;-1.6 100 10-> Take LatLon, Alt and Speed
    6 Static - 5 47.1;-1.5 160 0 -> Ignored because fields are already assigned
    = Resulting Pvt 1559876543 5 48.1;-1.6 100 10

    In order to allow monitoring of positioning reception and merging, the Pvt service displays the status of each provider (enabled, timedout) and their positioning data as source<n>.* MIB entries. The merged positioning data is displayed as pvt.* MIB entries.

    In order to emulate the GPS mode, the Pvt service compute a mode according to fields available in the merged Pvt data, regardless of the type of source (static, ntp, etc):

    • mode=1: time is available
    • mode=2: mode=1 + latlon is available
    • mode=3: mode=2 + alt is available

    Sample configuration file

    config/10-pvt.xml

    <?xml version="1.0" encoding="utf-8"?>
    <configuration>
    <service start="true" instance="0">Pvt</service>
    <config>
    <!-- Message Id for choir protocol-->
    <choir applicationid="20"
    vdpcacheid="0" />
    <!-- General configuration of the Pvt Service
    recomputeinterval: Interval to recompute the list of active sources when sources time out (ms)
    -->
    <pvtconfig recomputeinterval="2000" />
    <!-- List of sources (starting from 1)
    enabled: Whether this source is enabled
    profile: Profile of the source (autotalks/gpsd/native/time/user)
    name: Name of the source (for display)
    feedmsgid: Message ID to get data from
    timeout: Timeout to deprecate this source
    fields: Comma-separated list of fields to get from this source
    (time/ls2004/latlon/alt/speed/heading/climb)
    filtermode: Ignore messages whose GNSS mode is below filtermode
    fixspeedresetheadingthreshold: Reset heading if speed is below this value
    fixspeedupdateheadingthreshold: Update heading if speed is over this value
    -->
    <sources count="7" />
    <!-- The following comments describe the purpose of each pre-configured source -->
    <!--
    userdata: positioning data provided by the user
    (e.g. gps traces, external software, etc)
    No timeout is set because we want user input to last forever.
    Once the source is enabled and a fix has been received, it
    will stay as is until the source is disabled
    -->
    <source1 profile="user"
    name="userdata"
    enabled="false"
    feedmsgid="203"
    timeout="-1"
    filtermode="0" />
    <!--
    recorder: positioning data provided when replaying a record
    Not used yet, will be in a future version
    -->
    <source2 profile="native"
    name="recorder"
    enabled="false"
    feedmsgid="204"
    timeout="5000" />
    <!--
    autotalksprovider: positioning data provided by the Autotalks GNSS
    The field filtermode is set to 2 because time informations
    provided by Autotalks API are not accurate below mode 2.
    Fields fixspeedresetheadingthreshold and fixspeedupdateheadingthreshold
    are set because the heading is unaccurate at low speed.
    Needs to have the position service enabled in Autotalks.
    -->
    <source3 profile="autotalks"
    name="autotalksprovider"
    enabled="true"
    feedmsgid="201"
    timeout="2000"
    filtermode="2"
    fixspeedresetheadingthreshold="1.0"
    fixspeedupdateheadingthreshold="2.0" />
    <!-- staticprovider: static provision of leap seconds
    Provide leap seconds since 2004 from static configuration
    when the Autotalks chipset is not available. This is get
    from the staticprovider, but with fields limited to ls2004.
    Must be placed before gpsprovider and ntpprovider to make
    sure these modules will have leap seconds defined when
    performing time calculations. Otherwise, time calculations
    will be done with leap seconds set to 0.
    Needs to have the staticprovider service enabled.
    -->
    <source4 profile="native"
    name="staticprovider"
    enabled="true"
    feedmsgid="206"
    timeout="-1"
    fields="ls2004" />
    <!-- gpsprovider: positioning data provided by a GPSD daemon
    May be used if the hardware platform embeds a GNSS chipset
    supported by GPSD. May also be used to connect to an external
    GPSD server (for example for testing purpose).
    Needs to have the gpsprovider service enabled.
    -->
    <source5 profile="gpsd"
    name="gpsprovider"
    enabled="true"
    feedmsgid="205"
    timeout="2000" />
    <!-- ntpprovider: timing data provided from a NTP server
    May be used to retrieve the time information when no GNSS
    data is available.
    A timeout is set at a quite long time (30 minutes), however
    it is not really important since this is the last source for
    time data.
    Needs to have the ntpprovider service enabled.
    -->
    <source6 profile="time"
    name="ntpprovider"
    enabled="true"
    feedmsgid="207"
    timeout="1800000" />
    <!-- staticprovider: static positioning data
    May be used to define a static position for the ITS Station.
    It is put as the last module since one may want to use it as
    a fallback when other sources are unavailable. One may want
    to put it as the first source in order to overload every
    other sources.
    Note that the feedmsgid is not defined as the staticprovider
    is already present before in the list. There shall not be the
    same feedmsgid declared several times in the configuration.
    Needs to have the staticprovider service enabled.
    -->
    <source7 profile="native"
    name="staticprovider"
    enabled="true"
    timeout="-1" />
    <!-- feedmsgid="206" -->
    </config>
    </configuration>

    Auxiliary services

    The following services are not part of the Pvt service. However they are closely related to its functioning.

    StaticProvider

    The StaticProvider service publishes a static position as configured in its MIB. The fix is generated and sent at a configured rate defined by the config.refreshinterval MIB entry.

    Section Field R/W Type Default value Description
    choir messageid RW integer - Message ID to publish fixes
    config refreshinterval RW integer 1000 Rate to send the message (ms)
    staticposition latitude RW decimal - Static latitude (degrees)
    staticposition longitude RW decimal - Static longitude (degrees)
    staticposition posmajoraxisdirection RW decimal - Error direction (degrees)
    staticposition possemimajoraxiserror RW decimal - Major axis error (degrees)
    staticposition possemiminoraxiserror RW decimal - Minor axis error (degrees)
    staticposition altitude RW decimal - Static altitude (meters)
    staticposition altitudeerror RW decimal - Altitude error (meters)
    staticposition heading RW decimal - Static heading (degrees)
    staticposition headingerror RW decimal - Heading error (degrees)
    staticposition speed RW decimal - Static speed (m/s)
    staticposition speederror RW decimal - Speed error (m/s)
    staticposition climb RW decimal - Static vertical speed (m/s)
    staticposition climberror RW decimal - Vertical speed error (m/s)
    staticposition leapseconds2004 RW decimal - Leap seconds since 2004

    config/20-staticprovider.xml

    <?xml version="1.0" encoding="utf-8"?>

    <service start="false" instance="0">StaticProvider</service>

    latitude: static latitude of the station (deg)

    possemiminoraxiserror: static semi minor axis positioning error

    headingerror: static heading error (deg)
    climberror: static horizontal speed error (m/s)

    longitude="-1.5190593"
    altitude="182"
    speed="1.4"
    heading="189.2"
    climb="0"
    ls2004="5" />
    </config>
    </configuration>

    GpsProvider

    The GpsProvider service connects to a gpsd server to receive GNSS fixes from a GNSS chipset or an external server. Each time a GNSS fix is received from the server, the fix is forwarded to the Pvt service.

    Section Field R/W Type Default value Description
    choir messageid RW integer - Message ID to publish fixes
    gpsserver address RW string 127.0.0.1 Hostname/IP of the gpsd server
    gpsserver port RW integer 2947 Port of the gpsd server
    gpsserver retryinterval RW integer 3000 Interval to retry connection

    config/20-gpsprovider.xml

    <?xml version="1.0" encoding="utf-8"?>
    <configuration>
    <service start="false" instance="0">GpsProvider</service>
    <config>
    <!-- Message Id for choir protocol-->
    <choir messageid="204" />
    <!-- Configuration for gps provider
    address: address of the GPSD server
    port: port of the GPSD server
    retryinterval: interval for connection attempts
    -->
    <gpsserver address="127.0.0.1"
    port="2947"
    retryinterval="3000" />
    </config>
    </configuration>

    NtpProvider

    The NtpProvider service connects to a ntp server to synchronize time. The NtpProvider regurlarly refresh the time synchronization with the server. The time fix is then forwarded to the Pvt service.

    Section Field R/W Type Default value Description
    choir messageid RW integer - Message ID to publish fixes
    ntpserver address RW string 127.0.0.1 Hostname/IP of the ntp server
    ntpserver port RW integer 123 Port of the ntp server
    ntpserver retryinterval RW integer 3000 Interval to retry connection
    ntpserver requestinterval RW integer 3000 Interval to send requests

    config/20-ntpprovider.xml

    <?xml version="1.0" encoding="utf-8"?>
    <configuration>
    <service start="true" instance="0">NtpProvider</service>
    <config>
    <!-- Message Id for choir protocol-->
    <choir messageid="207" />
    <!-- Configuration for ntp provider
    address: address of the NTP server
    port: port of the NTP server
    retryinterval: interval for connection attempts
    requestinterval: interval between two requests
    -->
    <ntpserver address="0.pool.ntp.org"
    port="123" />
    </config>
    </configuration>

    PathHistoryManager

    The PathHistoryManager service computes the Path History from merged Pvt data pushed to the VDP cache. Then it pushes Path History points to the Path History cache. Points are computed according to the calculation method published in the VSCA standard.

    Section Field R/W Type Default value Description
    choir applicationid RW integer - App ID to register with
    choir vdpcacheid RW integer - Cache ID of VDP cache
    choir pathhistorycacheid RW integer - Cache ID of Path History cache
    pathhistory maxsize RW integer - Maximum size of Path History
    pathhistory allowableerror RW decimal 0 See standard
    pathhistory chordlengththreshold RW decimal 0 See standard
    pathhistory maxextimatedradius RW decimal 0 See standard
    pathhistory smalldeltaphi RW decimal 0 See standard
    pathhistory mindistance RW decimal 0 See standard

    config/20-pathhistorymanager.xml

    <?xml version="1.0" encoding="utf-8"?>
    <configuration>
    <service start="true" instance="0">PathHistoryManager</service>
    <config>
    <!-- Message Id for choir protocol-->
    <choir applicationid="21"
    vdpcacheid="0"
    pathhistorycacheid="9" />
    <!-- Path History parameters (see DOT HS 811 492B, Appendix B-2) -->
    <pathhistory maxsize="50"
    allowableerror="0.47"
    chordlengththreshold="22.5"
    maxextimatedradius="63781370"
    smalldeltaphi="1"
    mindistance="0" />
    </config>
    </configuration>

    PositionProvider

    A backward compatibility is provided for the legacy PositionProvider service configuration. If the PositionProvider is enabled and configured, its configuration is translated to the equivalent Pvt (and auxiliary services) configuration and these services are started by the PositionProvider.

    Usage examples

    Request UTC time

    $ mwconfig mib get -t Pvt pvt.now

    Output:

    pvt.now = 1554743227562713

    Note: This returns the current UTC time in milliseconds, computed on-demand by the Pvt service. It must be distinguished from utctime and itstime, which provide times at the reception of the lastest GNSS fix.
