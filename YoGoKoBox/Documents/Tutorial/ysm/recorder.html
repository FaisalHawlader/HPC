YoGoKo Smart Middleware

    Record/Replay

    The Recorder service allows to record messages passing via the middleware (e.g.

    sent messages, received messages, or GNSS fixes), to timestamp and store them into a MongoDB database. Recorded messages may then be replayed afterward.

    A record classically contains the journey of the ITS station along with arounding stations, events, traffic lights and signages.

    Recorder principles

    General principles

    The Recorder service allow user to capture messages on message IDs to save them into a MongoDB database. This way user can replay these messages at anytime to see what happened during his journey.
    The Recorder service has a status that defines the action in progress, this status could be:

    • 'stopped' : Recorder is waiting for an action.
    • 'recording': Recorder is recording messages.
    • 'replaying': Recorder is replaying messages.

    Recorder needs to be in 'stopped' mode to execute any action such as recording or replaying.

    When recording, Recorder will timestamped messages before inserting them into database. With this mechanism when Recorder will replay them, messages will be replay the same way as they were received.

    Recording messages

    The Recorder service can record messages published on message IDs as defined in its configuration file (80-recorder.xml).
    The Recorder service, to record, will need its status to be set to 'recording' mode and will also need a record name to identify this record in MongoDB database.
    When recording, it will subscribe to these message IDs, defaults are its own and incoming messages for CAM, DENM, IVI, SPAT and MAP.
    All messages received on a message ID will be timestamped and stored in a MongoDB database into a collection corresponding to this type of message.

    To stop recording, Recorder status should be set to 'stopped', then it will unsubscribe to these message IDs.

    Replaying messages

    The Recorder service can also replaying the records previously recorded.
    The Recorder service, to replay, will need its status to be set to 'replaying' mode and will also need the name of the record to replay.
    When replaying, it will publish messages stored in database to their corresponding message IDs, as if they were current messages.

    To stop replaying, Recorder status should be set to 'stopped', then it will stop publish to these message IDs.

    Some services may needs to be stopped before replaying a record, to avoid the messages currently received come bothered the replay.
    These services can be defined in mibs with ctrlservices and ctrlservice<n> sections.

    MIBs

    Details of MIB entries :

    Section Field R/W Type Default value Description
    choir applicationid RW int 601
    database server RW string Url of MongoDB server
    database dbname RW string Name of MongoDB database
    database capturelist RW string "capturelist" Name of the collection that keeps the references of all records
    recorder action W string Action in progress of Recorder service
    recorder status R string "stopped" Status of Recorder service
    recorder capturename RW string Name of record that is currently recorded/replayed
    recorder starttime R int 0 Timestamp in ms of when record/replay has begun
    recorder referencetime R int 0 Timestamp in ms, date of current record
    recorder elapsed R int 0 Elapsed time of current record in ms
    recorder duration R int 0 Duration of current record in ms
    msgids count RW int 0 Number of message IDs to subscribe
    msgid<n> id RW int 0 Message ID to subscribe to
    captures count R int 0 Number of records in MongoDB database
    capture<n> name R string Name of record
    capture<n> time R int 0 Timestamp date of record in ms
    capture<n> duration R int 0 Duration of record in ms
    ctrlservices count RW int 0 Number of services to disabled
    ctrlservice<n> servicename RW string Name of service to disabled
    ctrlservice<n> instanceid RW int 0 Instance ID of service to disabled
    ctrlservice<n> restartafterrecord RW bool true Restart service after a replay

    Usage examples

    Getting the Recorder status

    $ mwconfig mib get -t Recorder -i 0 'recorder.status'

    Output:

    recorder.status = "stopped"

    Note: If Recorder is already in "recording" or "replaying" mode, it is not possible to start recording

    Start Recording

    $ mwconfig mib set -t Recorder -i 0 'recorder.action="record"' 'recorder.capturename="Record_1"'

    Stop Recording/Replaying

    $ mwconfig mib set -t Recorder -i 0 'recorder.action="stop"'

    Start Replaying

    $ mwconfig mib set -t Recorder -i 0 'recorder.action="replay"' 'recorder.capturename="Record_1"'

    List records

    $ mwconfig mib get-all -t Recorder -i 0 |grep '^capture'

    Output:

    captures.count = 2
    capture1.name = "Record_0"
    capture1.time = 1539007571968
    capture1.duration = 254661
    capture2.name = "Record_1"
    capture2.time = 1540995685566
    capture2.duration = 223326
