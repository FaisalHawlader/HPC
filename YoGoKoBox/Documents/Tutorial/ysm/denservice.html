    Decentralized Environmental Notification Basic Service (DenService)

    The Decentralized Environmental Notification Messages (DENMs) are distributed within the ITS-G5 (802.11p) network and provide information related to a road hazard or an abnormal traffic conditions, such as its type and its position.

    A DENM is composed of one common ITS PDU header and multiple containers, which together constitute a DENM. The ITS PDU header is a common header that includes the information of the protocol version, the message type and the ITS-S ID of the originating ITS-S.

    Currently supported standard version:

    • ETSI EN 302 637-3 V1.2.2 (2014-11)
    • ETSI EN 302 637-3 V1.3.1 (2019-04)

    Processing and I/O

    On emission, DENM messages are triggered and generated by the DenService using informations contained in this trigger.

    On reception, DENM messages are passed to the upper layer, that is applications and services subscribed for reception of CAM messages.

    DenService

    Note: The DenService supports the following formats for emission and reception: ASN1_UPER, JSON, XML, Human Readable

    Triggering DENM messages

    The DenService will attempt to receive a Denm Trigger, Update or Termination on triggerdenm message id.

    A Denm trigger is used to indicate a new event. New DENMs will be generated.
    A Denm update allows the update of an existing event.
    A Denm termination indicates the end of an event.

    The trigger sent on triggerdenm message id contain all informations to generate a DENM and send it to outdenm message id If the handling of trigger, update or termination message succeed, management information are store into originating cache to handle the repetition transmission or the update of the message. The action id associated to the event will be return in a request answer to the application that request the trigger through PubSub request mechanism.

    The definitions of Denm trigger and trigger answer are described in the ASN.1 modules :

    • DENM-TRIGGER { iso (1) identified-organization(3) reserved (9900) yogoko (1) denmtrigger (2) version1 (1) revision(2) }
    • DENM-TRIGGER { iso (1) identified-organization(3) reserved (9900) yogoko (1) denmtrigger (2) version2 (2) revision(2) }

    The ASN.1 module to use for triggering event depend on the standard version used for the DenService.

    Standard version ASN.1 trigger module
    1.2.2 iso (1) identified-organization(3) reserved (9900) yogoko (1) denmtrigger (2) version1 (1) revision(2)
    1.3.1 iso (1) identified-organization(3) reserved (9900) yogoko (1) denmtrigger (2) version2 (2) revision(2)

    Denm triggers can be sent to the DenService either in ASN.1 UPER or JER.

    Emission of DENM messages

    The transmission processing will determine whether a received message on id outdenm should be sent or not to the V2X network. The processing is based on the station profile and the service configuration.

    • Timing : DENM emission depends on the event validity duration, the repetition duration and the repetition interval get from the trigger.
    • Security : If the security management is enabled, the service will check if the station are authorized to emit this kind of message or event.

    Once the transmission check succeed, message are sent over the itssocket and to the message id owndenm.

    The sending process will also attach the following information for the network and transport layer in metadata.

    Metadata Name Description
    trafficclass Traffic class that represents Facility-layer requirements on packet transport
    destinationarea_angleAngle of the geometric shape as defined in ETSI EN 302 931 in degrees from North.
    destinationarea_shape
    destinationarea_latitudeWGS 84 latitude for the centre position of the geometric shape as defined in ETSI EN 302 931 in 1/10 micro degree
    destinationarea_destaDistance a of the geometric shape as defined in ETSI EN 302 931 in metres.
    destinationarea_longitudeWGS 84 longitude for the centre position of the geometric shape as defined in ETSI EN 302 931 in 1/10 micro degree
    destinationarea_destbDistance b of the geometric shape as defined in ETSI EN 302 931 in metres.

    Reception of DENM messages

    DENM are received from an itssocket, The service will try to decode the ASN.1 message and failed if the received message is not encode with the current version implemented. If the recvcontrol.controlmsgs is enable, the service will check the expiration time of the message and drop it if is expire.

    If reception control succeed message are then sent to the indenm message id.

    Depending of the message values, the DenmService could forward the message and re-sent it to the V2X network layer

    Sample configuration file

    config/40-denservice.xml

    <?xml version="1.0" encoding="utf-8"?>
    <configuration>
    <!--
    DenService
    Provides support for ETSI DENM messaging
    -->
    <service start="true" instance="0">DenService</service>
    <config>
    <!-- Parameters for the pubsub mechanism (reasonable defaults) -->
    <pubsub senderid="91" receiverid="92" handlerid="93"
    indenm="45" outdenm="35" owndenm="65" triggerdenm="25" answerdenm="55" />
    <!-- Parameters for the cache mechanism (reasonable defaults) -->
    <cache originating="2" receiving="21" forward="22"/>
    <!-- Parameters for DENM emission through LTE-V2X media
    * enable : Whether the LTE-V2X media should be use for DENM emission.
    * msgtype : The string that identifies the ITS message type for LTE-V2X emission. This string can be used
    for example in the MQTT topic if MQTT protocol is used for the LTE-V2X emission.
    * repetitionthreshold : This attribute only concerns the repetition of a given DENM. A DENM must be repeated
    through LTE-V2X media if the last occurrence of the DENM that has been sent through LTE V2X exceeds
    repetitionthreshold value (in seconds) -->
    <ltev2x enable="false" msgtype="denm" repetitionthreshold="540" />
    <!-- Parameters for the itssocket mechanism (reasonable defaults) -->
    <!-- <socket receiver="gn+btp://cch:2002" sender="gn+btp://cch:2002" /> -->
    <socket denm="gn+btp://cch:2002" />
    <!-- Standard reference and version to use -->
    <standard reference="ETSI_EN_302637-3" version="1.2.2" />
    <!-- <standard reference="ETSI_EN_302637-3" version="1.3.1" /> -->
    <!-- Reception control parameters
    dropownmsgs: Whether the DenService should drop its own messages
    on reception
    controlmsgs: Whether the DenService should check messages on
    reception
    -->
    <recvcontrol dropownmsgs="false"
    controlmsgs="true"
    droprepetition="true"/>
    <!-- Security control parameters
    signmessages: Whether sent DENM messages should be signed
    securitycontrol: Whether the security service should control
    (lock) emission of DENM messages
    aid: AID of DENM messages for sent and received messages
    -->
    <securityparameters signmessages="false"
    securitycontrol="false"
    aid="37"/>
    <!-- DENM Keep-alive forwarding
    value: Enable DENM Keep-alive forwarding
    -->
    <forwarding value="false"/>
    <!-- Generation control parameters
    enabled: Lock station id during DENM emission
    -->
    <sendcontrol lockid="false"/>
    <!-- triggering forcing
    forceactionid: Allow applications to force their own ActionID in
    DENM triggers
    forcereferencetime: Allow application to force their own referenceTime in
    DENM triggers
    forcestationtype: Allow application to force their own stationType in
    DENM triggers
    -->
    <triggering forceactionid="false"
    forcereferencetime="false"
    forcestationtype="false"
    randomsequencenumber="false"/>/>
    <!-- DENM auto-update
    enabled: Automatically update DENMs close to expiration
    -->
    <autoupdate enabled="false"/>
    </config>
    </configuration>
