    Itsnet

    Overview

    itsnet implements the ETSI and IEEE network communication stack. It may be executed as a Linux daemon, as an Autotalks thread, or as a library.

    Changes in itsnet v2.4: From version v2.4, the Autotalks Craton1 build of libgn is not generated anymore, as the support of embedded libgn is dropped from YoGoKo Autotalks CR1 image. On such platforms, itsnet must be run as a Linux daemon and use V2X bridging to send/received frames to/from the radio. The present documentation page assumes that itsnet is running as a Linux daemon.

    Supported standards

    itsnet supports the following protocols:

    • GeoNetworking (ETSI EN 302 636-4-1 – v1.2.1)
    • GeoNetworking (ETSI EN 302 636-4-1 – v1.3.1)
    • BTP (ETSI EN 302 636-5-1 – v1.2.1)
    • BTP (ETSI EN 302 636-5-1 – v1.3.1)
    • GN6ASL (ETSI EN 302 636-6-1 – v1.2.1) - Only on Linux using TUN NF-SAP
    • Wave (IEEE P1609.3v3)
    • Security (version supported depends on the C-ITS Security Stack)

    Features

    Configuration modes

    itsnet may obtain its configuration either from a standalone configuration file or from YoGoKo Middleware through the Config mechanism.

    When the itsnet Linux daemon is executed in standalone configuration mode (-c flag), it directly parses the configuration file provided in argument, as part of its startup procedure. The configuration file is typically named /etc/itsnet/itsnet.conf. It is formatted under a key = value form with sections in brackets.

    When the itsnet Linux daemon is executed in mwconfig mode (-m flag), it connects to YoGoKo Middleware, declaring as a remote configurable service. The middleware then loads configuration to itsnet and controls the status of itsnet through the Config mechanism. In this mode, the configuration is typically read from XML files in the middleware repository /etc/mw-server/config/XY-itsnet.xml. The configuration file is formatted as described in the middleware documentation.

    -c and -m flags are not mutually exclusive. itsnet may thus retrieve configuration both from its configuration file and through the Config mechanism, in which case configurations acquired from the Config mechanism prevail. Moreover, some configuration parameters are currently only available in the itsnet.conf file.

    Configuration variables and sections may slightly differ between these modes, but are equivalent. Configuration variables are described in the Configuration section below.

    The variables defined in configuration control how itsnet operates at runtime.

    Specifically, some variables have to do with protocol defined values (timeouts, buffer sizes, etc) as well as interactions with external daemons. Protocol defined values are defaults fixed by the standard and, in general, should not be changed.

    Positioning source

    itsnet may acquire its positioning and timing from multiple sources:

    • From a static position declared in the configuration
    • From a gpsd server
    • From YoGoKo Middleware (choir protocol)

    The positioning source is declared and configured in itsnet configuration or through the Config mechanism under the position_service configuration block.

    Examples:

    • To use a static position (using the system clock as a time source) 
      position_service {
  type = "static"
  lat = 48.135288
  lon = -1.622724
  leap_seconds = 5
}
    • To connect to a gpsd server 
      position_service {
type = "gpsd"
host = "localhost"
port = "2947"
leap_seconds = 5
}
    • To connect to Y-Smart Middleware 
       position_service {
  type = "choir"
 }
      The position service will connect to the middleware address defined in the gn.choir section ('choir_target').

    Network protocol selection

    itsnet supports both ETSI Geonetworking and IEEE WAVE protocols.

    The configuration of the network protocol to use is done via the protocol.type and gn.protocol_version variables of the itsnet configuration file.

    protocol.type value gn.protocol_version value protocol activated
    GN 0 ETSI GN v1.2.1
    GN 1 ETSI GN v1.3.1
    WAVE - IEEE WAVE

    NF-SAP

    itsnet provides several NF-SAP to communicate with upper layers:

    • The TUN NF-SAP (Linux daemon only) creates a TUN interface that may be used to send/receive GN and GN6 packets
    • The Choir NF-SAP communicates with the upper layer through the choir protocol

    The nfsap to be used is controlled by the nfsap.type configuration variable. Each NF-SAP have its own configuration variable to control its behavior.

    Loopback

    itsnet provide a loopback feature using choir functionalities:

    • Encapsulated messages (with GN or WAVE layer) can be published on a specific message id
    • itsnet can also decoded encapsulated messages by publishing them on an other message id.

    GN6ASL

    GN6ASL (GeoNetworking to IPv6 Adaptation Sub-Layer) is a protocol that specifies the transmission of IPv6 packets over GeoNetworking. GN6ASL introduces the concept of a "geographical virtual link" (GVL) which is a virtual link spanning multiple physical links with geographical boundaries. In other words, it lets a station exchange IPv6 packets with nodes in a specific geographical area.

    GVLs can be statically defined by the configuration:

       gn6asl {
       config {
           enabled = true
           vi_resol_addr = 1
       }
       gvl {
           shape = "circle" # or "rect" or "elip"
           lat = 48.135288
           lon = -1.622724
           dist_a = 1000
           dist_b = 0
           angle = 0
       }
   }

    The configuration above enables GN6ASL and creates a single GVL in the form of a TAP interface, on which one can send IPv6 packets. Interfaces are named gvlX so as not to conflict with other tap interfaces that may exist on the system.

    Additional GVLs may be created at runtime if the station receives a GN-encapsulated Router Advertisement.

    Note: To use this feature, itsnet must be executed as a Linux daemon and configured to use the TUN NF-SAP.

    Configuration

    As stated above, itsnet may obtain its configuration from one of the following sources:

    • by reading its configuration from a standalone configuration file itsnet.conf (only when running as a Linux daemon)
    • by loading its configuration from YoGoKo Middleware configuration file XY-itsnet.xml

    Section mapping

    For historical reasons, configuration variables and sections are not the same in the standalone configuration file itsnet.conf and in the Middleware service configuration XY-itsnet.xml.

    The following list provides an indicative mapping between itsnet.conf sections (left) and XY-itsnet.xml configuration groups (right):

    • protocol -> protocol
    • gn.config -> gn
    • gn.network -> *(not present)*
    • gn.beacon_service -> beacon
    • gn.position_service -> position_service
    • gn.packet_forwarding -> forwarding
    • gn.location_service -> location
    • gn.choir -> choir
    • gnsec -> security
    • gnsec.choir -> choir
    • nfsap -> nfsap
    • loopback -> loopback

    Running itsnet

    As a Linux daemon

    Using systemd

    Start itsnet:

    $ systemctl start gn-itsnet

    Stop itsnet:

    $ systemctl stop gn-itsnet

    Restart itsnet:

    $ systemctl restart gn-itsnet

    Verify status:

    $ systemctl status gn-itsnet

    Manually

    $ itsnet -d -c /etc/itsnet/itsnet.conf

    Note: In order to run, itsnet requires either root privileges or CAP_NET_RAW capability, as it open raw socket on the network device.

    The -d command line switch keeps itsnet in the foreground. If it is left out, the daemon will send itself into the background.

    As an Autotalks thread

    Changes in itsnet v2.4: From version v2.4, the integration of libgn in Autotalks Craton 1 is not supported anymore.
