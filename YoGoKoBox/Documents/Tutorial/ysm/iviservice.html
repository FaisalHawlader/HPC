    Infrastructure to Vehicle Information Service (IviService)

    IVI service is one instantiation of the infrastructure services to manage the generation, transmission and reception of the IVI messages.

    An IVIM supports mandatory and advisory road signage such as contextual speeds and road works warnings. IVIM either provides information of physical road signs such as static or variable road signs, virtual signs or road works.

    The In Vehicle Signage Messages (IVIM) are distributed within the ITS-G5 (802.11p) network.

    An IVIM is composed of one common ITS PDU header and multiple containers, which together constitute an IVI. The ITS PDU header is a common header that includes the information of the protocol version, the message type and the ITS-S ID of the originating ITS-S.

    Currently supported standard version:

    • ETSI TS 103 301 v1.1.1 (2016-11)
    • ETSI TS 103 301 v1.2.1 (2018-08)*
    • ETSI TS 103 301 v2.1.1 (2021-03)

    *Because of binary compatibility between ASN.1 module from version 1.2.1 and from version 2.1.1, v1.2.1 and v2.1.1 based messages are also compatible. Due to this, if the service is configured with version 1.2.1, version 2.1.1 will be used for communication. The managment of sent/received messages depending on the configured version is detailed in the following table :

    Sent message \ Receiver configuration v1.1.1 v1.2.1 v2.1.1
    v1.1.1 fully decoded not decoded not decoded
    v1.2.1 not decoded fully decoded fully decoded
    v2.1.1 without extension not decoded fully decoded fully decoded
    v2.1.1 with extension not decoded fully decoded fully decoded

    Processing and I/O

    On emission, IVI messages are triggered and generated by the IviService using informations contained in this trigger.

    On reception, IVI messages are passed to the upper layer, that is applications and services subscribed for reception of IVIM messages.

    IvIService

    Note : The IviService supports the following formats for emission and reception: ASN1_UPER, JSON, XML, Human Readable

    Triggering IVI messages

    The IviService will attempt to receive an Ivi Trigger, Update or Termination on triggerivi message id.

    A Ivi trigger is used to indicate a new signage. New IVI messages will be generated.
    A Ivi update allows the update of an existing signage.
    A Ivi termination indicates the end of an signage.

    The trigger sent on triggerivi message id contains all informations to generate an IVI message and send it to outivi message id If the handling of trigger, update or termination message succeed, management information are store into originating cache to handle the repetition transmission or the update of the message. The action id associated to the event will be returned in a request answer to the application that request the trigger through PubSub request mechanism.

    The definition of Ivi trigger and trigger answer are described in the ASN.1 module IVI-TRIGGER { iso (1) identified-organization(3) reserved (9900) yogoko (1) ivitrigger (2) version1 (1) }.

    Ivi triggers can be sent to the IviService either in ASN.1 UPER or JER.

    Emission of IVI messages

    The transmission processing will determine whether a received message on id outivi should be sent or not to the V2X network. The processing is based on the station profile and the service configuration.

    • Timing : IVI message emission depends on the signage validity duration, the repetition duration and the repetition interval retrieved from the trigger.
    • Security : If the security management is enabled, the service will check if the station is authorized to emit this kind of message or signage.

    Once the transmission check succeeds, message are sent over the itssocket and to the message id ownivi.

    The sending process will also attach the following information for the network and transport layer in metadata.

    Metadata Name Description
    trafficClass Traffic class that represents Facility-layer requirements on packet transport
    destinationarea_angleAngle of the geometric shape as defined in ETSI EN 302 931 in degrees from North.
    destinationarea_shape
    destinationarea_latitudeWGS 84 latitude for the centre position of the geometric shape as defined in ETSI EN 302 931 in 1/10 micro degree
    destinationarea_destaDistance a of the geometric shape as defined in ETSI EN 302 931 in metres.
    destinationarea_longitudeWGS 84 longitude for the centre position of the geometric shape as defined in ETSI EN 302 931 in 1/10 micro degree
    destinationarea_destbDistance b of the geometric shape as defined in ETSI EN 302 931 in metres.

    Reception of IVI messages

    IVI messages are received from an itssocket. The service will try to decode the ASN.1 message and will return an error if the received message is not encoded with the version currently implemented. If the recvcontrol.controlmsgs is enabled, the service will check the expiration time of the message and drop it if is expired.

    If reception control succeeds message are then sent to the inivi message id.

    Sample configuration file

    config/40-iviservice.xml

    <?xml version="1.0" encoding="utf-8"?>
    <configuration>
    <!--
    IviService
    Provides support for ETSI IVIM messaging
    -->
    <service start="true" instance="0">IviService</service>
    <config>
    <!-- Parameters for the pubsub mechanism (reasonable defaults) -->
    <pubsub senderid="107" receiverid="108" handlerid="97"
    inivi="37" outivi="47" ownivi="67" triggerivi="27" answerivi="57" />
    <!-- Parameters for the cache mechanism (reasonable defaults) -->
    <cache originating="4" receiving="41" stationPropertiesId="3" />
    <!-- Parameters for DENM emission through LTE-V2X media
    * enable : Whether the LTE-V2X media should be use for DENM emission.
    * msgtype : The string that identifies the ITS message type for LTE-V2X emission. This string can be used
    for example in the MQTT topic if MQTT protocol is used for the LTE-V2X emission. -->
    <ltev2x enable="false" msgtype="ivi" />
    <!-- Standard reference and version to use -->
    <standard reference="ETSI_TS_103301" version="2.1.1" />
    <!-- <standard reference="ETSI_TS_103301" version="1.1.1" /> -->
    <!-- Reception control parameters
    dropownmsgs: Whether the IviService should drop its own messages
    on reception
    controlmsgs: Whether the IviService should check messages on
    reception
    -->
    <recvcontrol dropownmsgs="true"
    controlmsgs="false"/>
    <!-- Security control parameters
    signmessages: Whether sent IVIM messages should be signed
    securitycontrol: Whether the security service should control
    (lock) emission of IVIM messages
    aid: AID of IVIM messages for sent and received messages
    -->
    <securityparameters signmessages="false"
    securitycontrol="false"
    aid="139" />
    <!-- IVIM auto-update
    enabled: Automatically update IVIMs close to expiration
    -->
    <autoupdate enabled="false"/>
    </config>
    </configuration>
