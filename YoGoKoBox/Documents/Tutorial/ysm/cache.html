YoGoKo Smart Middleware

    Cache

    The Cache mechanism manages a remotely accessible cache, allowing applications to get, set or delete values, or to be notified when the cache is altered by another application.

    Cache

    The cache mechanism is organized as follows:

    • Each cache is independent: identified by a "cache id"
    • Each cache contains a list of objects: identified by an "object id"
    • Each object is a key->value dictionary, where the key is a string
    • Values can be of the following types: boolean, integer, double, string, byte array
    Cache design

    Available commands

    Command name Description
    List List objects in a cache
    Get Retrieve an object from a cache
    Set/Replace Modify an object in a cache
    Remove Delete an object from a cache
    Register Register to modifications performed on an object

    List command

    List available objects for given cache

    I/O Mand. Variable Description
    in x Cache Id id of the cache to query, e.g. the station's own data (from GNSS, CAN bus, I/O) is stored in cache 0.
    out x Object(s) list List of available objects for this cache id

    }
    }

    Get command

    Get content (a dictionary of key/value) of the whole object or only the specified keys.

    I/O Mand. Variable Description
    in x Cache Id id of the cache to query
    in x Object Id id that identifies an object, e.g. for VDP there is only one object (1), for the stations's Local Dynamic Map (Cache 10) each object represent a unique station
    in Key list List of requested keys in object, leave empty to retrieve whole object
    out Object The requested object

    Code samples

    get : get VDP cache

    Command Line

    $ mwcache -a 12345 -c 0 -o 1 -g lat lon

    C/C++ API

    // Build a list with the needed keys.
    char* keys[] = {"lat","lon"};
    int key_count = 2;
    // Get vdpcache with the keys list.
    struct choir_map* cache_object = choir_cache_get(choir_handler, 0, 1, key_count, keys);
    // Get latitude and longitude from returned cache_object and print them.
    double latitude = choir_map_get_double(cache_object, "lat", 0);
    double longitude = choir_map_get_double(cache_object, "lon", 0);
    printf("latitude : %f\n", latitude);
    printf("longitude : %f\n", longitude);

    Java API

    // Get instance of the CacheService

    Set Command

    Update the object identified by objectId in a given cache. If the object does not exist, a new one is created.

    Individual values can be deleted at the same time by setting their value to "null". Other key/value pairs are kept untouched.

    All registered clients will receive a notification indicating which event happened (CREATE or UPDATE).

    I/O Mand. Variable Description
    in x Cache Id id of the cache to query
    in x Object Id id that identifies an object, e.g. for VDP there is only one object (1), for the stations's Local Dynamic Map (Cache 10) each object represent a unique station
    in x object Key-value pairs to set in the object

    Code samples

    set: on a station configured as a vehicle set vehicleRole=3 (dangerousGood)

    Command Line

    $ mwcache -a 12345 -c 0 -o 1 -s 'vehiclerole=3'

    C/C++ API

    struct choir_map * IDCacheObject = choir_map_init(1);
    choir_map_put_int(IDCacheObject, "vehiclerole", 3);
    // Apply to vdp cache (ID 0).
    choir_cache_set(choir_handler, 0, 1, IDCacheObject);

    Java API

    // Get instance of the CacheService
    CacheService cacheService = yogokoClient.getCacheService();
    // Set vehiclerole to 3 (dangerousGood)
    cacheService.set(0, 1, "vehiclerole", 3);

    Websocket API (JS)

    Replace Command

    Replace an object with a new one. If no object with the given object id exists, a new one is created.

    All registered clients will receive a notification indicating which event happened (REPLACE or CREATE).

    I/O Mand. Variable Description
    in x Cache Id id of the cache to query
    in x Object Id id that identifies an object, e.g. for VDP there is only one object (1), in the stations's Local Dynamic Map (Cache 10) each object represents a unique station
    in x object The new object

    Code samples

    replace: Replace whole VDP content (cache id = 0, object id = 1).

    Command Line

    $ mwcache -a 23232 -s -c 0 -o 1 'lat=48.1350568' 'lon=-1.6221928' 'speed=5' 'heading=90'

    Register command

    The register command may be used by an application to receive cache object(s) periodically or when an object has been edited. The application must register to a cache id and may filter on a specific object or keys in it The application may register to be notified periodically or when one of the following events occur:

    • CREATE: The object has been created.
    • UPDATE: At least one key-value of the object has been updated.
    • REPLACE: Object content has been flushed and re-createed.
    • DELETE: The object has been deleted.
    • UNKNOW: The register method has been called to notify at a fixed frequency.
    I/O Mand. Variable Description
    in x Cache Id Target cache id.
    in x Object Id Target object id or "null" to register to all objects
    in Frequency Frequency in milliseconds between notifications or 0 to only be notified on content edit
    in Key list List of keys of interest, user will only be notified on a edit of those keys
    out x register id Registration id to use to unregister

    The register callback method is defined as follows:

    I/O Mand. Variable Description
    in x type The event type CREATE/UPDATE/REPLACE/DELETE/UNKNOW
    in x register id The id that identifies the registration
    in x object id The id of the dictionary affected by the event
    in x keyChangedMap List of keys that have been modified
    in x Cache Object The object that has been updated, key-value pairs

    Sequence diagrams

    Synchronous cache access (set/get)

    Cache set/get

    Message distribution

    1. Data producer sets a key->value pair for the object identified by objectId in the cache identified by cacheId
    2. Data user gets the value for the key in the object identified by objectId in the cache identified by cacheId
    3. Cache Service returns the corresponding value
    4. Data producer removes the object identified by objectId in the cache identified by cacheId
    5. Data user gets the value for the key in the object identified by objectId in the cache identified by cacheId
    6. Cache Service returns no value because the object no longer exists

    Note:

    • Data producer and data user may address an arbitrary objectId in an arbitrary cacheId. If the cacheId or objectId does not exist, it is created.

    Asynchronous cache access (set/notify)

    Cache set/notify

    Message distribution

    1. Data user registers to a cache identified by cacheId
    2. Data producer sets a key->value pair for the object identified by objectId in the cache identified by cacheId
    3. Cache Service notifies the data user of an object being modified, and forwards the modified object
    4. Data producer removes the object identified by objectId in the cache identified by cacheId
    5. Cache Service notifies the data user of an object being deleted

    Note:

    • In addition to the cacheId, registration may be performed on a specific objectId and/or a specific list of keys to monitor
    • Instead of notifying the user on modification, Cache Service may notify at at given frequency
